## 架构分层（已确认）

```
用户需求
    ↓
Task Pool (任务层入口)
    ↓
Thinking Pool (拆解需求 → 原子任务规格)
    ↓
POST Runtime (批次扫描与投递)
    ↓
Construct Runtime (层内搬运、field 落位、派发、进程/信号管理)
    ↓
constructor Agent (具体架构分析与工单撰写)
    ↓
Gate Pool (监督 + 路由，待实现)
    ↓
Work Pool (执行 → 写入统一项目根)
    ↓
产物输出
```

### 关键设计原则
- **Thinking 只负责拆解**：输出任务意图和说明，不负责项目落地目录
- **POST 负责完整性投递**：等待整个 batch 完成后再投递到 Construct
- **Construct Runtime 只负责运送与管理**：负责 field 落位、任务派发、槽位/进程/信号/超时管理
- **constructor Agent 负责具体执行**：负责读取 batch 输入并实际产出 `work_task_*.txt`
- **Work 直接写入项目根**：不再各自在独立 workspace 产出
- **Outbox 只记录证据/摘要/必要结果**：不承担项目文件主存储

## Construct Pool 当前实现细节

### Runtime 已确认实现的能力
- POST 可将 Thinking Outbox 的 txt 或目录型产物投递到 `pools/construct/Queue/`
- ConstructRuntime 在 `list_queue_tasks()` 中先执行 `_preprocess_queue_folders()`：
  - 识别 Queue 中的目录项
  - 提取 `BATCH_ID`（优先读 `summary.txt`，否则回退目录名）
  - 将目录内容移动到 `pools/construct/fields/{batch_id}/input/`
  - 自动生成 `task_batch_{batch_id}.txt` 引用任务
  - 写入 `meta/batch_info.json`
- Runtime 负责：
  - 派发任务到 constructor 槽位
  - 生成 launch bat
  - 管理进程生命周期
  - 接收并处理 terminal signals
  - 超时回收与 artifact 收集
  - 在 batch 终态后清理对应 field 目录

### 不属于 Runtime 的能力
- Runtime **不直接**做业务架构分析
- Runtime **不直接**写出最终 `work_task_*.txt`
- Runtime **不直接**决定 `TARGET_FILE`、`METHOD`、类签名等工程细节
- 上述具体操作由 constructor Agent 在运行时完成

### 当前运行链路
- Constructor 接收 `task_batch_{batch_id}.txt`
- constructor Agent 从 `BATCH_FIELD/input/` 读取 summary 与 task 文件
- constructor Agent 在 `BATCH_FIELD/output/` 产出 `work_task_*.txt`
- Runtime 在终态时自动收集 workspace 产物到 Outbox
- 若任务为 `batch_*`，Runtime 完成后会自动清理对应 `fields/{batch_id}/`

### 关键保障
- 派发锁：`threading.RLock()` 避免并发重复派发
- `.processing` 改名占位，避免同一 queue 文件被重复消费
- 派发失败回滚：恢复 queue 文件并清理 slot 污染
- fallback done：Claude 进程提前退出时由 bat 尾部强制补发 done 信号
- timeout 收敛：超时写 terminal event、回收进程、保留产物、释放槽位
